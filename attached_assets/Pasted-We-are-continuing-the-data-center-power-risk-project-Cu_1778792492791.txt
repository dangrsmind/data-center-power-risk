Add a guarded, explicit promotion UI on /project-candidates.

Do not auto-promote anything.

Task:
Update the /project-candidates page so an analyst can promote one project candidate at a time through an explicit confirmation flow.

Requirements:

1. Inspect existing frontend and backend:
- frontend/src/pages/ProjectCandidatesPage.tsx
- frontend/src/api/adapter.ts
- frontend/src/api/types.ts
- backend/app/api/routes/project_candidates.py
- backend/app/schemas/project_candidate.py

2. Add frontend API adapter function.

Add a function such as:

promoteProjectCandidate(candidateId, options)

It should call:
POST /project-candidates/{candidate_id}/promote

Request body should include:
{
  "confirm": true,
  "allow_unresolved_name": false,
  "allow_incomplete": false
}

Expose optional booleans for:
- allow_unresolved_name
- allow_incomplete

3. Update types.

Add TypeScript types for:
- promotion request
- promotion response

Response fields likely include:
- dry_run
- promoted
- promoted_project_id
- project_created
- project_updated
- evidence_created
- warnings
- errors
- would_create_project
- would_promote
- would_update_project

Match the backend response shape.

4. Add visible row action.

On /project-candidates, each row should have a read-only action set plus a guarded promotion action:
- Open source
- Copy URL
- Details
- Promote

Promotion action rules:
- Show Promote only for candidates that are not already promoted.
- For already promoted candidates, show clear label:
  Promoted
  and show/copy promoted_project_id if available.
- Do not allow bulk promotion.

5. Confirmation flow.

Clicking Promote should open a confirmation modal/panel, not immediately call the API.

The modal/panel should show:
- Candidate name
- State/county/city if present
- Primary source URL
- Confidence
- Source count
- Claim count
- Evidence excerpt
- Clear warning:
  “This will create a real Project record from this candidate. Continue only after review.”
- Checkbox or explicit confirmation text/button.

The final button should say:
Promote candidate

6. Safety options.

If candidate_name starts with “Unresolved” or is missing:
- Show warning that unresolved names are normally blocked.
- Include an explicit checkbox:
  Allow unresolved name

If state or other required fields are missing:
- Show warning.
- Include an explicit checkbox:
  Allow incomplete candidate

The API call should pass those booleans.

7. Result handling.

After promotion API response:
- If errors are returned, show them clearly and do not mark as promoted.
- If promoted=true and promoted_project_id is returned:
  - update the row locally or refetch /project-candidates
  - show success message
  - show promoted_project_id
  - row should display Promoted after refresh/refetch
- Show warnings returned by backend, for example candidate_already_promoted.

8. UI clarity.

Keep consistent dark theme and readable text.
Promotion controls must be visually distinct but not dangerous-looking.
Use clear labels.
Do not hide row actions behind hover-only UI.
Keep the table layout compact; do not reintroduce large column gaps.
Actions column should remain easy to find.
